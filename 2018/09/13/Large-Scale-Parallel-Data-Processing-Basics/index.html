Large-Scale Parallel Data Processing Basics

Posted on 2018-09-13 | In Big Data | Visitors:

What are speedup and scaleup, and how do we measure them for a given application?

Scaleup is the ability to sustain the same performance levels (response time) when both workload and resources increase proportionally:

Speedup = sequentialTime / parallelTime

Note: to determine speedup, the work to be done is fixed!

On n identical machines, the theoretically best possible speedup is n.

Speedup is the effect of applying an increasing number of resources to a fixed amount of work to achieve a proportional reduction in execution times:

Scaleup = workDoneParallel / workDoneSequential

Note: to determine scaleup, the time to work on the job is fixed!

On n identical machines, the theoretically best possible scaleup is n.

Is load balance a measure of good parallelization?

load imbalance = $\frac{load \;on \;the \;most \;loaded \;machine}{avg \;load \;over \;all \;machines}$

which is a secondary metric. When the total remains same, smaller as it as better.

Will adding more processors always speed up computation? Why or why not?

No, adding too many processors might lead to cost of resources for communication and scheduling.

Adding more compute power does not speed up computation, when data transfer to the processors is the bottleneck.

What does Amdahl’s Law say? Give an example to illustrate it.

Amdahl’s law is a formula which gives the theoretical speedup in latency of the execution of a task at fixed workload that can be expected of a system whose resources are improved.

Amdahl’s law is often used in parallel computing to predict the theoretical speedup when using multiple processors.

For example, if a program needs 20 hours using a single processor core, and a particular part of the program which takes one hour to execute cannot be parallelized, while the remaining 19 hours (p = 0.95) of execution time can be parallelized, then regardless of how many processors are devoted to a parallelized execution of this program, the minimum execution time cannot be less than that critical one hour. Hence, the theoretical speedup is limited to at most 20 times (1/(1 − p) = 20). For this reason, parallel computing with many processors is useful only for highly parallelizable programs.

Amdahl’s law can be formulated in the following way:

$$ S_{latency}(s) = \frac{1}{(1-p)+\frac{p}{s}} $$

where

  • $S_{latency}$ is the theoretical speedup of the execution of the whole task;
  • s is the speedup of the part of the task that benefits from improved system resources;
  • p is the proportion of execution time that the part benefiting from improved resources originally occupied.

Furthermore,

$ S_{latency}(s) <= \frac{1}{1-p}$
equals to $\frac{1}{1-p}$ when s approaches infinity.

Why is programming with shared data structures generally not considered “scalable”?

Sychronization. In order to guarantee thread security, we need to apply locks on data structure. When it’s scaling there will be lots of operations of lock/unlock, which would affect the performance and become not scalable.

  1. Concurrent access to shared data structures requires careful synchronization through locking.
  2. Locking reduces parallelism and increases computation cost for lock management.
  3. The more workers we use for increased parallelism, the higher the probability of locking conflicts. For this reason, the shared-memory programming approach is generally not considered highly scalable.

Show an example of a “concurrency bug” where concurrent access to a shared data structure causes inconsistency.

Concurrency bugs usually happen, when two or more things occur at the same time in a place which wasn’t designed to work in a concurrent environment.

Probably the most common type of concurrency problem that everyone who starts working with concurrent programming stumbles upon is a race condition. It usually manifests itself as an unpredictable and particularly difficult to debug erroneous behavior, which happens because each of the workers supposes that no other workers which work with the same resources exist at that time. What happens is some worker might make changes that other workers aren’t aware of and as a result, invalid behavior might occur from one of the workers.

Why is it beneficial to have data partitioned across many machines before the computation starts?

Could explain it use the idea of divide and conquer. After partition data to machines, the program could be run simultaneously on machines, which could accelerate computation.
